# `kaname` ビジネスルール定義書 (business-rules.md)

## 1. 起動周期およびサーバーレススケジューリングルール

### GCPベストプラクティスに基づくアーキテクチャ

- 起動周期的スケジュール管理は、アプリケーションのプログラムコード内（cronデーモン等）で行うことを厳禁とする。
- トリガー制御は、GCP Cloud Scheduler で定義された cron パターンに基づき稼働する。
- 実行時のみコンテナを立ち上げてジョブを実行する GCP Cloud Run Jobs をトリガー対象とすることで、1回あたりの収集コストを極小化しつつ、時次・日次・週次などのスケジュール頻度変更をシステム実装の変更なし（完全疎結合）に運用する。
- **インフラタイムアウトによるコスト暴走防止ルール：**
  エージェントのバグや通信のハングアップによる無限ループ・コスト高騰をインフラ層で機械的に遮断するため、GCP Cloud Run Jobs の最大実行時間（タイムアウト）は「30~60分（想定最大処理時間に安全マージンを付加した値）」を上限として厳格に設定・デプロイしなければならない。


## 2. トピック格納中間ディレクトリの自動分類ロジック

トピックMarkdownファイルが単一ディレクトリ直下にフラットに過剰に蓄積されるのを防ぎ、かつ過度なフォルダの乱立を抑えるための階層分類ルール。

### 分類基準のメタデータ参照ルール

- 提案エージェント（Aegis-Writer）は、新規トピックを生成する際、対象トピックが属する情報源（SSoT）のコンテキスト情報、および抽出したプロパティ（YAML Frontmatterの tags や category 等）を参照する。
- エージェントはトピックを「大分類（中間ディレクトリ）」のフォルダへと割り振る。
  - （例：NCOのトピック ⇒ `topics/gov-agencies/NCO.md`、JVN脆弱性情報 ⇒ `topics/vulnerabilities/CVE-xxxx.md`）

### フォルダ乱立の最大制限ルール（最大100フォルダ）

- 中間ディレクトリの総数が乱立し混乱するのを防ぎ、かつ過度なオーバーヘッドを避けるため、中間フォルダの最大上限数は「100以下」とシステムレベルで厳格に制約を課す。
- 新しいカテゴリフォルダを自律作成する際、既存のフォルダ数がすでに「95」に達している場合は、新規フォルダ作成を行わず、意味的に最も類似する既存フォルダ（例：`topics/misc/` や `topics/others/` などの共通の受け皿フォルダ）へ格納することを義務付ける。これにより、ナレッジベース全体の階層のフラットさと秩序性を両立する。


## 3. 動的キーワード抽出および既存ファイルのインクリメンタルアップデートルール

### 抽出基準の非ハードコーディング方針

- 抽出対象は「専門用語」などの固定カテゴリーに限定しない。
- 提案エージェントへのシステムプロンプト内に、情報源（SSoT）メタデータに付随する「関心ターゲット記述」を動的に埋め込み、そのドメインコンテキストに最適化された重要トレンドキーワードをLLMに自律判別させる。

### インクリメンタル・アップデート（既存ファイルの更新）ルール

- 提案エージェントは、新規キーワードに関するファイルを書き出す前に、すでに同一名（または名寄せ対象）のMarkdownファイルが Obsidian Vault 内に存在するかを確認する。
- 既存ファイルが存在する場合、エージェントはそのMarkdownの全内容をコンテキストとして読み込む。
- エージェントは、既存のテキスト構造（H1, H2などの見出し）を破壊することなく、新しく検知された追加ファクト・最新動向を、論理的な新セクション（H3など）の挿入、または既存文脈の改訂という形で「追記・統合」する。
- 完全な上書きや既存ファクトの損失は、査読エージェントによる検証時にリジェクト対象とする。

### 孤立トピック自動接続ロジック（Orphan Note Resolution）

- 提案エージェントは、ドキュメント生成・更新ループの最終フェーズにおいて、Vault内にあるすべてのMarkdownファイルから「リンク流入（Inbound Link）」および「リンク流出（Outbound Link）」がゼロである孤立トピックファイルを洗い出す。
- 洗い出された孤立ノートについて、エージェントは既存の主要なトピック群とのセマンティックな関係性を推論する。
- 推論結果に基づき、関連性が最も高い既存トピックドキュメントの解説文章内に、自然な文脈として孤立トピックへの内部リンクを動的に追記更新してコミットし、ナレッジグラフ全体のネットワーク密度を自律的に最大化する。


## 4. レポート更新差分比較および内部リンク参照ルール

### 「直近の収集差分」の定義と入力処理

- クロール処理の開始時に、リポジトリ内（レポート用ディレクトリ等）から最新の日付スタンプを持つレポートMarkdownファイルを1件ロードする。
- 現在クロールした最新コンテンツ（テキストおよびメタデータ情報）を、ロードした直近レポートの内容と比較する。

### 内部リンク参照による重複記述の抑制ロジック

- 直近レポートや既存Wikiトピックに記述されている内容と意味的に同一・類似した事実をレポートに再出力することは禁止とする。
- 記述を最小化するため、提案エージェントは差分レポート生成時に、既存トピックや過去レポートの該当ページへの内部リンクを動的に生成して埋め込む。これにより、ナレッジベース全体のファイル間の接続密度を最大化しつつ、レポートの紙幅（新規追加事項）を最小に抑える。


## 5. マルチエージェント自律マージおよびレビュー制御ルール

### マルチエージェント意思決定チェーン

- **提案フェーズ（Aegis-Writer）：**
  提案エージェントが新規トピック作成、既存トピック更新、孤立ノートの解決、および差分要約レポートの執筆を実行。インテリジェンス専用ブランチ（`osint/content-*`）を作成しコミット。GitHub MCPを介して main へのPRを起票。
- **自動レビューフェーズ（Aegis-Reviewer）：**
  査読エージェントが、起票されたPRの内容（diff）およびGitHub ActionsのCI結果（リンク切れ、OFM書式整合性テスト、および Takumi Guard による脆弱性パス結果）を取得する。
- **合意形成およびマージ：**
  査読エージェントは、更新内容が「開発綱領のOFM規約、上書き禁止ポリシー、リンク形成ポリシー」をすべて遵守しているかを検証する。問題がなければ Approve コメントを残し、GitHub MCP経由で main への自律マージを実行する。
- **フィードバックループ：**
  検証ルールに抵触していた場合、査読エージェントはPRコメントに具体的なリジェクト理由を書き込み、マージを行わない。提案エージェントがそれを読み、修正コミットを作成する。規定回数（デフォルト：3回）を超えてもマージ合意に至らない場合は、自律的に管理者に差し戻し（Issue起票）を行う。


## 6. Discord通知トリガー制御ルール

### 通知チェーンの厳格化

- PR作成時、または main マージ時のプッシュイベントを直接のDiscord通知契機として用いることを禁止する。
- DiscordへのWebhook送信は、必ず本番環境（Cloudflare Pages）への静的ファイルの展開完了を知らせる `pages-deployment` イベント（またはActions上のデプロイ成功確認ステップ）のフックに限定する。これにより、サイトにアクセスできないリンクの誤通知を100%防止する。


## 7. 障害起票および通知ルール

### GitHub Issueネイティブ通知の適用

- クローリングの連続失敗回数が規定数に達した際、システムはGitHub MCPを用いてIssueを起票する。
- 本システム自体にはメールのSMTPプロトコルやサードパーティ製配信サービスを一切実装しない。管理者はGitHub側の「Watch」および「Notifications」設定を利用して、自律起票されたIssueをメール、Slack統合、モバイルアプリ等で自律的に受け取るものとする。
