# `kaname` 品質検証チェックリスト (checklist.md)

## 1. 開発綱領（constitution.md）適合性監査

### [ ] 先進的ビルド・トランスパイル環境の検証

- パッケージマネージャーに pnpm が採用され、`pnpm-lock.yaml` を用いて決定論的なインストールが保証されていること。
- 開発およびテスト時に tsx による瞬時ロードが働き、プロダクションビルド時において esbuild が極小バンドルを出力できていること。

### [ ] 完全サーバレス・ゼロコスト待機アーキテクチャの検証

- 常時起動型のVMインスタンス（Compute Engine等）が排除されていること。
- GCP Cloud SchedulerおよびGCP Cloud Run Jobsを組み合わせた、実行時課金のみのステートレスバッチ構成がマニフェストレベルで維持されていること。
- GCP Cloud Run Jobs の設定において、無限ループやハングアップ時のコスト暴走を機械的に遮断するため、最大実行時間（タイムアウト）が30~60分程度に明示的に絞り込まれていること。

### [ ] AIモデルロックインの排除

- Aegis-Client 抽象ラッパー層を仲介し、コンフィグ変更のみで最新のフラグシップモデルAPIエンドポイントへ切り替え可能な設計が維持されていること。

### [ ] フロントエンド・ビルドパフォーマンス保護

- Quartz v5の設定において、Graph View（ローカルおよびグローバル）が完全に無効化（disabled）されていること。

### [ ] ブランチ・デプロイ運用制限の検証

- main ブランチへの手動での直接のPushおよびマージが、GitHubリポジトリの設定（Branch Protection Rules）でブロックされていること。
- 自律コミットがインテリジェンス専用プレフィックス（`osint/*`）に制限されていること。
- Cloudflare Pagesによる本番自動ビルド・デプロイが main マージ時のみトリガーされる構成になっていること。


## 2. セキュリティ ＆ サプライチェーン脆弱性監査

### [ ] Takumi Guard による依存モジュール保護監査

- GitHub ActionsのPR自動テストラインに Flatt Security社提供の Takumi Guard が正しく組み込まれていること。
- 脆弱性やライセンス違反が検出されたPRのビルドおよび Aegis-Reviewer によるApproveプロセスが機械的に100%遮断・防御されること。

### [ ] ゼロ・シークレットポリシーの検証

- ソースコード、環境設定、マニフェストファイル、GitHub Actionsの設定内にAPIキーや認証トークン（PAT）が直接記述されていないこと。
- すべての秘匿情報が GCP Secret Manager からロードされ、実行時に動的注入されること。

### [ ] GitHub App 認証セキュリティの検証

- 個人アクセストークン（PAT）を使用せず、GCP上の実行環境からGitHub Appの秘密鍵を用いて、有効期限1時間未満の一時インストーショントークンを動的生成・使用する構成であること。


## 3. ナレッジマネジメント（LLM Wiki）整合性監査

### [ ] トピック中間ディレクトリ自動分類と上限クォータ監査

- 前処理されたメタデータおよびタグ（YAML Frontmatterのタグ・SSoT ID等）に基づき、論理的な「大分類中間フォルダ」へファイルが自動振り分けされていること。
- 既存中間フォルダ総数が「95」に達した際、新規フォルダ作成が自動で抑制され、代替フォルダ（`topics/misc/`等）へ格納される上限保護が正しく機能していること。全体フォルダ数が100を決して超えないこと。

### [ ] 不変・可変データ特性（ミュータビリティ）の二相分離制御

- `ssot.yml` および過去の日付スタンプ付き「定時レポートMarkdown」が完全不変（Immutable）として保護されていること。
- トピック/概念/組織の解説Markdownが最新ファクトのマージ・名寄せ・アップデートを許容する可変（Mutable）として保護されていること。

### [ ] 上書き禁止（インクリメンタルアップデート）の遵守

- 既存トピックファイルが存在する場合、LLMがファイルを単純上書き（ファクト消去）せず、既存テキストを保持しながら差分セクションを追記統合していること。

### [ ] セマンティックリンク密度（Orphan Note Resolution）の検証

- 収集プロセス完了時に被リンク数ゼロの孤立ノートが検出された際、関連性の最も高い既存ドキュメント内に内部リンクが自然に追記更新されること。

### [ ] 重複排除と相互内部リンクによるサマリー最小化の検証

- レポート作成時、過去のレポートや既存Wikiに記載済みの内容の再記述を排除し、`[[内部リンク]]` を能動的に用いて参照させることで、文章量はコンパクトに抑えながら接続密度を高めていること。


## 4. 自律オーケストレーション ＆ フォールバック監査

### [ ] マルチエージェント意思決定チェーンの監査

- 提案エージェント（Aegis-Writer）によるPR作成後、査読エージェント（Aegis-Reviewer）がPR DiffおよびCI成否を自律評価し、Approveコメントおよび自律マージを完結できること。

### [ ] 無限マージループおよび暴走防止の検証

- 査読・提案間のフィードバック修正ループが最大3回でハードリミット制限されていること。
- ループ上限超過、または重大なクローリング失敗発生時、マージ処理を停止し、管理者向けの「GitHub Issue」を自律的に自動起票して安全にフォールバックすること。

### [ ] Discord誤送信防止の検証
- DiscordへのWebhook通知トリガーが、マージイベントではなく「Cloudflare Pagesの本番デプロイ成功イベント」に厳格にバインドされており、不正リンク・破損リンクの公開を100%防止していること。
